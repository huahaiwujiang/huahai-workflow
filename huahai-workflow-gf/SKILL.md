# Git Flow（gf）

自动化的 git 工作流程管理技能，简化日常代码提交操作。

---

## 流程概览

执行顺序：**状态检查 → 暂存 → 提交 → 拉取 → 推送**

**智能模式规则：**
- 状态检查、暂存、提交、拉取、推送：自动执行，无需询问
- 提交信息：自动根据变更内容生成，不再询问用户
- 拉取冲突：暂停，提示用户手动解决

---

## 执行流程

### 1. 检查状态（自动）

```bash
git status
```

**处理逻辑：**
- 如果有未跟踪文件，显示文件列表
- 如果工作区干净，显示当前分支信息，终止流程
- 如果有已修改但未暂存的文件，列出修改内容

### 2. 暂存文件（自动）

**暂存前检查（.gitignore 同步）：**

如果 .gitignore 中有规则但对应文件仍被 git 跟踪，先从缓存中移除：

```bash
# 检查是否有已被跟踪但应被忽略的文件
git ls-files -i --exclude-from=.gitignore

# 如果有，从缓存中移除（不删除本地文件）
git rm --cached <上述列出的文件>
```

然后再暂存所有变更：

```bash
git add -A
```

**说明：** 暂存后再次运行 `git status` 确认暂存状态。

### 3. 提交（自动）

**直接根据变更内容自动生成提交信息并执行，不询问用户。**

先查看变更详情，然后生成合理的提交信息：

```bash
# 查看变更内容（用于生成提交信息）
git diff --cached --stat

# 自动提交
git commit -m "<自动生成的提交信息>"
```

**提交信息生成规则：**

根据变更文件和内容，按以下规则生成：

| 变更类型 | 提交格式 | 示例 |
|---------|---------|------|
| 新增功能文件 | `feat: <功能描述>` | `feat: 新增用户登录模块` |
| 修复 bug | `fix: <修复描述>` | `fix: 修复登录验证失败问题` |
| 文档变更 | `docs: <文档描述>` | `docs: 更新 API 文档` |
| 代码重构 | `refactor: <重构描述>` | `refactor: 优化查询性能` |
| 样式调整 | `style: <样式描述>` | `style: 调整按钮样式` |
| 测试相关 | `test: <测试描述>` | `test: 添加登录单元测试` |
| 配置/依赖 | `chore: <配置描述>` | `chore: 更新依赖版本` |
| 小改动/测试 | `chore: <描述>` | `chore: 测试空行添加` |

**生成逻辑：**
1. 查看变更文件路径，判断变更类型
2. 如果是单个文件，提取文件名或功能名
3. 如果是多个文件，归纳整体变更目的
4. 生成简洁、准确的提交信息
5. 直接执行提交，输出提交结果给用户

**提交信息格式要求：**
- subject：不超过 50 字符，中文/英文均可，结尾不加句号
- body（可选）：解释 what 和 why，每行不超过 72 字符
- 使用祈使语气（"添加" 不是 "添加了"）

**Commit 粒度规则：**
- 一个 commit = todolist.md 中的一个子任务
- 变更文件数：1-8 个，代码行数：50-300 行
- ⚠️ 超过 500 行应拆分多个 commit
- 每个 commit 必须能编译通过

### 4. 拉取远程更新（自动）

```bash
git pull origin <当前分支>
```

**冲突处理：**

如果拉取时发生冲突：
1. 显示冲突文件列表
2. 暂停流程
3. 等待用户处理

### 5. 推送到远程（自动）

```bash
git push origin <当前分支>
```

**说明：** 推送前确保拉取已完成且无冲突。

## 特殊情况处理

### 无变更时

如果 `git status` 显示工作区干净，提示用户：当前暂无变更需要提交，然后终止此次流程。

### 分支不存在

如果目标分支不存在：
1. 显示可用分支列表
2. 询问用户选择正确的分支

## 注意事项

- 确保在 git 仓库目录下执行
- 确保有远程仓库连接
- 暂存前自动同步 .gitignore，确保被忽略的文件不被提交
