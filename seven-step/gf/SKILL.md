# Git Flow（gf）

一个智能的 git 工作流程管理技能，简化日常代码提交和分支合并操作。

---

## 流程概览

执行顺序：**状态检查 → 暂存 → 提交 → 拉取 → 推送 → 合并**

**智能模式规则：**
- 状态检查、暂存、提交、拉取、推送：自动执行，无需询问
- 提交信息：自动根据变更内容生成，不再询问用户
- 拉取冲突：暂停，提示用户手动解决
- 合并：始终询问确认

---

## 执行流程

### 1. 检查状态（自动）

```bash
git status
```

**处理逻辑：**
- 如果有未跟踪文件，显示文件列表
- 如果工作区干净，显示当前分支信息
- 如果有已修改但未暂存的文件，列出修改内容

### 2. 暂存文件（自动）

根据状态结果，自动暂存：

```bash
# 暂存所有变更（推荐）
git add -A

# 或按用户偏好暂存特定文件
git add <具体文件>
```

**说明：** 暂存后再次运行 `git status` 确认暂存状态。

### 3. 提交（自动）

**直接根据变更内容自动生成提交信息并执行，不询问用户。**

先查看变更详情，然后生成合理的提交信息：

```bash
# 查看变更内容（用于生成提交信息）
git diff --cached --stat

# 自动提交
git commit -m "<自动生成的提交信息>"
```

**提交信息生成规则：**

根据变更文件和内容，按以下规则生成：

| 变更类型 | 提交格式 | 示例 |
|---------|---------|------|
| 新增功能文件 | `feat: <功能描述>` | `feat: 新增用户登录模块` |
| 修复 bug | `fix: <修复描述>` | `fix: 修复登录验证失败问题` |
| 文档变更 | `docs: <文档描述>` | `docs: 更新 API 文档` |
| 代码重构 | `refactor: <重构描述>` | `refactor: 优化查询性能` |
| 样式调整 | `style: <样式描述>` | `style: 调整按钮样式` |
| 测试相关 | `test: <测试描述>` | `test: 添加登录单元测试` |
| 配置/依赖 | `chore: <配置描述>` | `chore: 更新依赖版本` |
| 小改动/测试 | `chore: <描述>` | `chore: 测试空行添加` |

**生成逻辑：**
1. 查看变更文件路径，判断变更类型
2. 如果是单个文件，提取文件名或功能名
3. 如果是多个文件，归纳整体变更目的
4. 生成简洁、准确的提交信息
5. 直接执行提交，输出提交结果给用户

**提交信息格式要求：**
- subject：不超过 50 字符，中文/英文均可，结尾不加句号
- body（可选）：解释 what 和 why，每行不超过 72 字符
- 使用祈使语气（"添加" 不是 "添加了"）

**Commit 粒度规则：**
- 一个 commit = todolist.md 中的一个子任务
- 变更文件数：1-8 个，代码行数：50-300 行
- ⚠️ 超过 500 行应拆分多个 commit
- 每个 commit 必须能编译通过

### 4. 拉取远程更新（自动）

```bash
git pull origin <当前分支>
```

**冲突处理：**

如果拉取时发生冲突：
1. 显示冲突文件列表
2. 暂停流程
3. 询问用户：「检测到拉取冲突，请手动解决冲突后回复「是/y/yes」，我将从头开始流程」

**用户回复处理：**
- 回复「是/y/yes」→ 从步骤1重新开始
- 回复其他 → 等待用户处理

### 5. 推送到远程（自动）

```bash
git push origin <当前分支>
```

**说明：** 推送前确保拉取已完成且无冲突。

### 6. 合并分支（询问确认）

**合并流程：**

1. 显示当前分支信息
2. 询问用户确认：「是否确认从 **dev** 合并到 **test**？（回复 yes/y/是 立即合并，回复 no/n/不是 选择其他目标分支）」

**用户回复处理：**

| 用户回复 | 操作 |
|---------|------|
| yes / y / 是 | 执行 `git checkout test` → `git merge dev` → 推送 test |
| no / n / 不是 | 询问：「请提供要合并到的目标分支名称」→ 用户回复后执行合并 |

**合并命令序列：**

```bash
# 切换到目标分支
git checkout <目标分支>

# 拉取目标分支最新代码
git pull origin <目标分支>

# 合并源分支
git merge <源分支>

# 推送合并结果
git push origin <目标分支>

# 切回原分支
git checkout <原分支>
```

## 特殊情况处理

### 无变更时

如果 `git status` 显示工作区干净，提示用户：当前暂无变更需要提交，然后终止此次流程。

### 分支不存在

如果目标分支不存在：
1. 显示可用分支列表
2. 询问用户选择正确的分支

### 合并冲突

合并时发生冲突：
1. 显示冲突文件
2. 提示用户：「合并发生冲突，请手动解决后回复「完成」」
3. 等待用户解决并回复
4. 执行 `git add` 和 `git commit` 完成合并

## 快捷用法

用户可以使用以下方式触发：

| 用户输入 | 执行操作 |
|---------|---------|
| `/gf` | 执行完整流程 |
| `git flow` | 执行完整流程 |
| `提交代码` | 执行到推送为止 |
| `合并分支` | 只执行合并步骤 |
| `发布到test` | 执行合并到test |

## 示例对话

核心交互模式（详细规则见上方各步骤）：

- **正常流程**：自动执行状态→暂存→提交→拉取→推送，然后询问合并确认
- **拉取冲突**：暂停，提示用户手动解决，回复「是/y/yes」后从头开始
- **其他目标分支**：合并询问时回复 no，然后输入目标分支名称
- **合并冲突**：显示冲突文件，提示用户解决后回复「完成」

---

## 注意事项

- 确保在 git 仓库目录下执行
- 确保有远程仓库连接
- 合并前确保代码已提交
- 合并后自动切回原工作分支
